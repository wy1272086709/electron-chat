# 登录后首条消息长时间转圈问题修复

## 1. 问题现象

用户刚登录后立即进入私聊或群聊并发送普通文本消息，消息可以先显示在界面中，但气泡外侧的发送状态圆圈偶尔很久才消失。

CSS 动画本身固定为 `0.8s/圈`，不存在随机降速。真正变慢的是消息从 `pending` 进入 `sending/sent` 的时间，UI 在这些状态下复用了同一个转圈图标。

## 2. 状态含义

```text
pending  -> 等待 Socket 可用并进入发送队列
sending  -> 已通过 Socket 发出，等待服务端 ack
sent     -> 服务端确认成功，移除转圈
failed   -> 超时或服务端明确拒绝，显示失败标记
```

因此“转圈很久”可能发生在两个阶段：等待 Socket 连接，或等待服务端 ack。第一轮修复解决了 Socket 初始化和历史同步阻塞；本轮继续解决服务端已经落库并广播消息、但 ack 没有及时回调时，文本消息仍固定等待的问题。

## 3. 根因

### 3.1 pending 发送被全量同步阻塞

旧的 Socket `connect` 回调顺序是：

```text
Socket connected
  -> 刷新会话列表
  -> 逐个会话同步历史消息
  -> 所有同步完成
  -> flush pending 消息
```

逐个同步使用串行 `await`。当用户会话较多、历史接口较慢或网络有波动时，刚发送的文本消息虽然已经写入本地可靠队列，却必须等待所有会话同步完成后才能真正发出。

会话数量越多，等待时间越明显，所以问题表现为“偶现”。

### 3.2 登录阶段可能重复建立 Socket

`currentUserId` 初始值为 `null`，用户信息从安全存储异步读取。旧 Socket effect 在身份读取完成前就可能启动连接；随后 `currentUserId` 更新，effect 清理旧连接并重新建立连接。

旧时序可能是：

```text
Socket A 开始连接
  -> 用户身份读取完成
  -> effect 清理 Socket A
  -> Socket B 重新连接
  -> pending 消息继续等待
```

如果用户刚好在切换窗口发送消息，就会看到较长的等待圆圈。

### 3.3 ACK 和消息广播不是同一个信号

服务端处理一条消息时，可能有两个独立结果：

1. 调用 Socket.IO ack callback，直接回复发送方。
2. 通过 `message:new` 向房间广播服务端消息。

实际运行中可能出现“消息已成功落库并通过 `message:new` 广播，但 ack callback 没有及时调用”的情况。接收方已经看到消息，发送方却仍在等待 ack。

旧文本逻辑使用 `socket.timeout(8000)`，一次最多等待 8 秒；超时后还会以同一个 `clientMessageId` 重试，最多 3 次。因此动画没有卡住，而是业务状态确实长时间停留在 `sending`。

登录后的首条消息更容易暴露该问题，因为 Socket 刚建立、房间同步和事件监听都处于初始化高峰，ack 与广播到达时间差更明显。

## 4. 解决方案

### 4.1 身份初始化门禁

新增 `authInitialized` 状态：

```text
读取用户和可靠队列状态
  -> 写入 currentUserId
  -> authInitialized = true
  -> 创建 Socket
```

Socket effect 在 `authInitialized=false` 时直接返回，避免使用未初始化身份建立临时连接，也避免 `currentUserId` 更新造成登录阶段的无效重连。

### 4.2 消息优先于历史同步

新的 Socket `connect` 顺序是：

```text
Socket connected
  -> 立即 schedulePendingFlush
  -> 后台刷新会话列表
  -> 后台逐个同步历史消息
```

pending 队列不再依赖全量同步完成。用户刚发送的消息会在 Socket 可用后立即进入发送流程。

历史同步仍然执行，用于补拉离线消息和校准本地状态，但不再位于发送关键路径上。

### 4.3 ACK、广播和短宽限窗口并行判定

文本和媒体消息现在使用相同的 1.5 秒确认窗口：

- 收到成功 ack：立即使用服务端消息结束本地状态。
- 收到失败 ack：立即进入失败状态。
- 收到包含相同 `clientMessageId` 的 `message:new`：服务端消息替换本地乐观消息，转圈立即消失。
- 1.5 秒内没有 ack，也没有及时回推：先结束 UI 转圈，再立即执行当前会话增量同步。
- 同步仍未找到服务端消息：可靠队列继续保留该消息，8 秒后使用相同 `clientMessageId` 在后台校验重试。
- 后台校验期间 UI 保持 `sent`，不会反复重新显示转圈；连续 3 次仍无法确认时才显示失败标记。

`clientMessageId` 仍然保留，用于服务端幂等、实时广播去重和同步补拉去重。宽限窗口结束时只改变 UI 状态，不立即删除可靠队列；只有 ACK、实时回推或同步确认服务端消息后才移除队列项。

### 4.4 为什么不再固定等待 8 秒

当前用户体验问题的关键是：8 秒超时并不能证明消息失败，只能证明 ack 没有到达。服务端消息可能已经落库并广播，因此继续显示发送中反而是不准确的状态。

新方案把明确失败 ACK 作为失败依据，把成功 ACK、`message:new` 和增量同步都作为成功校准来源。短宽限窗口只控制 UI 等待时间，不改变消息的幂等 ID，也不提前删除可靠队列。

## 5. 修复后时序

### 登录后立即发送文本

```text
身份初始化完成
  -> Socket 单次建连
  -> 用户点击发送
  -> 消息乐观上屏（pending）
  -> Socket connect
  -> 立即 flush pending
  -> sending
  -> 服务端 ack 或 message:new
  -> sent，转圈消失

后台并行：刷新会话和同步历史消息
```

### 已在线发送文本

```text
点击发送 -> pending -> sending -> ack/message:new -> sent
```

如果回执和广播都暂时未到：

```text
点击发送
  -> pending
  -> sending
  -> 等待最多 1.5 秒
  -> UI 标记 sent
  -> 立即增量同步当前会话
  -> 找到服务端消息：校准本地消息并清理可靠队列
  -> 未找到：保留队列，8 秒后后台校验重试（UI 不重新转圈）
```

## 6. 最终效果

- 登录阶段不再创建并切换两条 Socket 连接。
- 首条消息不再等待全部会话历史同步。
- 正常网络下，ACK 或服务端回推到达时，文本消息转圈立即消失。
- 后端不及时 ACK 时，转圈最长约 1.5 秒；必要的多轮校验转入后台，不再持续占用 UI 发送状态。
- 会话数量不会继续线性增加首条消息等待时间。
- 私聊、群聊、文本和媒体统一使用相同的确认规则。
- `clientMessageId`、本地可靠队列、实时回推去重和增量同步继续保留。
- 1.5 秒宽限结束不会直接丢弃队列消息，后台仍能发现真正未落库的异常。
- 服务端明确返回失败时，仍然显示失败标记。

如果 Socket 尚未连接或网络断开，消息仍会停留在 `pending`，直到连接恢复。这属于真实的未发送状态，不受 1.5 秒确认窗口影响。

## 7. 验证场景

- 登录后立即发送第一条普通文本消息。
- 登录拥有大量历史会话的账号后立即发送消息。
- 已登录且 Socket 在线时连续发送多条文本。
- 服务端成功落库并广播、但不调用 ack callback，确认转圈约 1.5 秒内消失。
- 服务端正常返回成功 ack，确认转圈立即消失。
- 暂时屏蔽 ack 和 `message:new`，确认 UI 结束转圈后可靠队列仍保留，并在后台使用相同 ID 校验。
- 断网发送后恢复网络，确认 pending 消息自动发出。
- 服务端明确返回失败，确认显示失败标记。
- 私聊和群聊分别验证一次。
- 发送文件和图片，确认上传完成后外侧转圈及时消失。
