# IPC 背压与限流说明

## 背景

当前项目通过 `ipcRenderer.invoke('fetch-data')` 把后端 HTTP 请求交给主进程代理。这个模型本身有回执：渲染进程会等待主进程返回 Promise 结果。

如果短时间内大量 UI 刷新、WebSocket 事件、轮询或重复点击同时触发 API 请求，主进程可能堆积过多未完成请求，表现为：

- 主进程内存上涨
- 后端请求瞬间放大
- 渲染进程等待时间变长
- 未读数/通知刷新出现抖动

因此需要做请求级背压。

## 当前实现

主进程 `fetch-data` IPC 增加了有界并发队列：

- 同时执行请求数默认：`8`
- 等待队列长度默认：`80`
- 队列满时立即返回：

```json
{
  "result": false,
  "data": null,
  "message": "请求过于频繁，请稍后重试",
  "code": 429
}
```

可通过环境变量调整：

```bash
ELECTRON_IPC_MAX_CONCURRENT=8
ELECTRON_IPC_MAX_QUEUE_SIZE=80
```

## 为什么先不做分片

当前 `fetch-data` 传输的是请求配置和 JSON 响应，属于小对象 IPC。分片主要适用于：

- 大文件
- 大图片或视频二进制
- 超大历史消息导出
- 大量记录一次性跨进程传输

普通 API 请求默认不分片。对 API 请求做分片反而会增加协议复杂度，而且不能解决并发请求过多的问题。

## 回执限流

`ipcRenderer.invoke` 已经具备请求-响应回执语义。当前策略是：

1. 渲染进程发起 IPC 请求。
2. 主进程进入并发队列。
3. 主进程完成 HTTP 请求后返回结果。
4. 队列满时直接返回 `429`，让调用方知道当前需要降速。

这就是当前项目的回执限流模型。

## 429 前端处理

统一请求层会处理 `code === 429`：

- 展示轻提示：`操作太频繁，请稍后重试`
- 提示做 3 秒节流，避免一波并发请求导致多个提示连续出现
- 原始响应仍会返回给业务调用方，业务层可以按 `result/code/message` 决定是否做额外处理

## 渲染层降压

前端也做了两类降压：

1. `secureStorageService` 增加窗口内内存缓存，避免每个 API 请求前都 IPC 读取 token。
2. WebSocket 触发的会话/通知刷新会在短时间内合并，连续事件只触发一次 `GET /chat/rooms` 或 `GET /notifications`。

## 什么时候需要消息队列

当前队列是进程内内存队列，适合 UI 请求背压。

如果未来出现以下场景，再考虑持久化消息队列：

- 离线发送消息，需要断网后恢复发送
- 大批量上传/下载任务
- 后台同步任务需要跨应用重启恢复
- 需要严格顺序、重试、去重和状态追踪

## 什么时候需要分片或流式传输

如果后续要跨 IPC 传输大文件或大二进制，不要走 `fetch-data` JSON 代理。建议：

- 文件选择只通过 IPC 返回文件路径
- 文件上传交给主进程或专门 upload IPC
- 大文件使用流式读取/上传
- 大列表使用分页，不一次性跨进程传输

## 后续建议

- 对按钮点击类请求增加 UI loading，避免重复提交。
- 对搜索输入增加 debounce。
- 对 WebSocket 高频事件保持合并刷新，不要每个事件都直接拉接口。
- 对高价值写操作可在业务层补充禁用按钮或 loading，减少触发 `429` 的概率。
